<objective>
Add a minimal persistence and quiz-lifecycle layer so successful quiz generations are stored locally and can be retrieved later, while leaving provider wiring, quality validation, and the thin UI trigger untouched.

Purpose: extend the app with local quiz history and session retrieval built strictly on top of the public `generateQuiz` boundary, without introducing auth, analytics, scoring, or external infrastructure.
Output: a small quiz session model, a local persistence adapter, an app-layer lifecycle service, and tests proving quiz storage and retrieval behave safely.
</objective>

<modules>
- `src/app/quiz-session.ts`: canonical stored quiz/session types, including topic, question set, answers, explanations, timestamps, and stable identifiers.
- `src/app/quiz-store.ts`: a minimal repository port plus a simple local persistence implementation that can be backed by a JSON file or an in-memory test double.
- `src/app/quiz-lifecycle.ts`: orchestration service that calls `generateQuiz`, stores successful sessions, and exposes list/get operations for past quizzes.
- `src/app/index.ts`: public export surface for generation and lifecycle helpers.
- `tests/app/quiz-session.test.ts`: verifies stored session shapes and serialization rules.
- `tests/app/quiz-store.test.ts`: verifies repository behavior, ordering, and round-trip persistence.
- `tests/app/quiz-lifecycle.test.ts`: verifies successful generations are stored and retrievable, while failures do not create history entries.
</modules>

<data_flow>
1. The caller enters through the public app boundary and requests generation via `generateQuiz`.
2. A successful generation result is transformed into a persisted quiz session snapshot containing topic, questions, correct answers, explanations, and timestamp.
3. The lifecycle service stores that snapshot through the repository port.
4. Retrieval APIs read the stored session list or a single session by id and return app-safe data only.
5. Generation failures or storage failures are surfaced as app-safe results and do not leak provider, config, or transport internals.
</data_flow>

<storage_strategy>
- Use a minimal local persistence layer with a single repository interface and a file-backed JSON store as the default implementation.
- Keep the file format append-friendly and atomic on write so a partial save cannot corrupt the history file.
- Preserve a small in-memory adapter for tests and for environments where file persistence is undesirable.
- Store only quiz-session data and lifecycle metadata; do not store provider responses, config payloads, or transport details.
- Treat the storage layer as local single-user state, not as a multi-user or synchronized data store.
</storage_strategy>

<access_layer_design>
- Define a `QuizPersistencePort` with explicit methods for `saveSession`, `listSessions`, and `getSessionById`.
- Keep lifecycle orchestration in `quiz-lifecycle.ts`, where `generateQuiz` is composed with the persistence port.
- Expose retrieval helpers through the app barrel so callers stay on the public app boundary.
- Normalize all stored and retrieved records into typed `QuizSessionSummary` and `QuizSessionDetail` shapes.
- Ensure storage failures return stable app-safe errors instead of provider, filesystem, or serialization details.
</access_layer_design>

<scope_guardrails>
- No auth.
- No analytics.
- No scoring engine.
- No multi-user system.
- No provider architecture changes.
- No quality validation changes.
- No UI trigger or render helper changes.
- No retries or fallback logic.
</scope_guardrails>

<recommended_implementation_order>

</recommended_implementation_order>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `generateQuiz` result -> persistence layer | Only validated, app-safe quiz results may be stored; provider/config/transport details must not cross this boundary. |
| persistence layer -> retrieval API | Stored sessions must be read back as typed quiz-session records, not raw storage blobs. |
| retrieval API -> caller | Callers can only see quiz content and stable lifecycle metadata, never storage internals. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Tampering | `src/app/quiz-store.ts` | mitigate | Validate stored session shapes on read and write, and use atomic replacement for file-backed persistence. |
| T-05-02 | Information Disclosure | `src/app/quiz-lifecycle.ts` | mitigate | Persist only quiz content and lifecycle metadata; return app-safe errors that hide provider and filesystem details. |
| T-05-03 | D | local history growth | accept | The feature is single-user and local-only, so bounded local history growth is acceptable for this phase. |
</threat_model>
