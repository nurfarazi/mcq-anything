<objective>
Create the first application-level workflow on top of the completed LLM foundation so the app can accept a topic and question count, generate MCQs, validate the output, and return only app-safe results.

Purpose: satisfy the phase-2 generation flow while keeping the app provider-neutral, fail-closed on malformed model output, and ready for a future UI or CLI to consume.
Output: a small app-layer orchestration path, strict app-side validation helpers, a minimal entry point, and tests that prove the happy path plus safe failure handling.
</objective>

- `src/app/quiz-types.ts`: define the app-facing request, result, and error contracts used by the workflow. Keep these types free of provider internals.

<data_flow>

1. A caller sends `{ topic, questionCount }` to the app entry point.
2. The entry point normalizes and validates the inputs before generation begins.
3. The workflow uses only `src/llm/index.ts` to resolve the active provider and create the provider implementation.
4. The provider returns a normalized MCQ response through the public LLM boundary.
5. The app validates the returned MCQs before exposing them to the caller.
6. Any failure is converted into an app-safe error result; provider internals never escape the app layer.

</data_flow>

<validation_strategy>

- Validate request inputs first: the topic must be non-empty after trimming, and `questionCount` must be a positive integer within a small v1 limit.
- Validate output second: every MCQ must have a non-empty question, exactly four options, a valid correct-answer index, and a non-empty explanation.
- Fail closed on malformed output; do not partially surface questions that failed validation.
- Keep validation in the app layer so provider adapters stay provider-neutral and reusable.

</validation_strategy>

<app_safe_error_handling>

- Return a discriminated union or equivalent app result that separates success from failure cleanly.
- Use stable error codes such as `INVALID_INPUT`, `GENERATION_FAILED`, and `INVALID_MODEL_OUTPUT`.
- Strip provider URLs, adapter details, and stack traces from caller-visible failures.
- Preserve the underlying cause for internal diagnostics without leaking it through the public workflow boundary.

</app_safe_error_handling>

<scope_guardrails>

- No authentication or user accounts.
- No persistence, saved history, or databases.
- No scoring, grading, or review modes.
- No analytics.
- No additional providers or provider-specific business logic branches.

</scope_guardrails>

<recommended_implementation_order>

1. Shared app contracts for requests, results, and error codes.
2. Input normalization and bounds checking for topic and question count.
3. Output validation for the normalized MCQ response.
4. The quiz-generation use case over the public LLM boundary.
5. The minimal app entry point.
6. End-to-end happy-path and failure-path tests.

</recommended_implementation_order>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Caller input -> app entry point | Topic and question-count inputs are untrusted and must be normalized before generation begins. |
| App -> public LLM boundary | Normalized provider responses are still untrusted until the app validates them. |
| App -> caller result | Internal failures must be translated into app-safe errors so provider details do not leak. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | T | `src/app/quiz-input.ts` | mitigate | Validate and clamp topic/questionCount before any provider call so malformed input cannot drive unexpected generation behavior. |
| T-02-02 | Tampering | `src/app/quiz-validation.ts` | mitigate | Reject malformed or incomplete MCQ payloads with fail-closed validation instead of passing broken content to the caller. |
| T-02-03 | Information Disclosure | `src/app/quiz-generation.ts` | mitigate | Map provider and transport failures to app-safe error codes/messages and keep endpoint/adapter details out of caller-visible output. |
| T-02-04 | D | entry point limits | mitigate | Enforce a small v1 question-count ceiling so the app cannot be pushed into unexpectedly expensive generation requests. |
</threat_model>
