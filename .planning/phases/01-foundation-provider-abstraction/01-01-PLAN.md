<objective>
Create the provider-neutral foundation for MCQ generation so the rest of the app can ask for AI output without knowing which backend is active.

Purpose: satisfy the phase-1 requirement set by making provider choice configuration-driven, defaulting to LM Studio, and keeping provider-specific behavior behind a stable abstraction.
Output: a shared LLM provider interface, env-driven provider resolution, an LM Studio adapter, and tests that prove provider swapping does not alter business logic.
</objective>

<modules>
- `src/llm/types.ts`: define the provider interface, neutral request/response shapes, and the strict MCQ schema contract used across providers.
- `src/llm/config.ts`: read environment/config and resolve the active provider key, with LM Studio as the default.
- `src/llm/providers/lm-studio.ts`: translate the neutral request into the LM Studio request format and map the response back into the shared contract.
- `src/llm/provider-factory.ts`: expose the single entry point that returns the configured provider implementation.
- `src/llm/index.ts`: re-export only the public abstractions so application code never imports a concrete provider directly.
</modules>

<data_flow>
1. Application code asks the provider factory for the active provider.
2. The factory reads config/env and chooses the provider key, defaulting to LM Studio.
3. The active provider receives a neutral generation request and returns a normalized response.
4. Business logic consumes only the shared response shape, never the provider-specific payload.
5. Runtime validation hooks are defined at the contract boundary so future phases can reject malformed model output without changing provider callers.
</data_flow>

<provider_abstraction_design>
- Keep the provider interface narrow: one generation method, neutral inputs, and a normalized output type.
- Hide all request URL, headers, authentication, and payload differences inside the concrete provider adapter.
- Resolve providers through config only; no `if provider === ...` branching in feature code.
- Make LM Studio the default implementation, but treat it as just one adapter behind the interface.
- Shape the abstraction so a future OpenAI provider can be added by implementing the same interface and extending config mapping only.
</provider_abstraction_design>

<validation_strategy>
- Define the strict MCQ schema in the shared types so every provider targets the same contract.
- Validate provider configuration at startup or provider resolution time so unknown provider keys fail fast.
- Treat malformed provider responses as a safe error path rather than trying to silently repair them.
- Reserve full output-shape enforcement for the later output-validation phase, but make the boundary explicit now so the contract cannot drift.
</validation_strategy>

<risks_and_edge_cases>
- LM Studio may be unavailable or misconfigured locally; provider resolution should fail clearly and not fall back to a different backend silently.
- A model may ignore the requested question count or return text that is not valid MCQ structure.
- A future provider may support different request options, so the abstraction must avoid leaking provider-specific fields into application code.
- Configuration mistakes can make the app appear to work while actually pointing at the wrong backend; defaults and logging must make the active provider obvious.
</risks_and_edge_cases>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User input -> generation request | Topic and question-count inputs are untrusted and must be constrained before reaching the provider layer. |
| App -> LM Studio HTTP API | Local model responses are untrusted until they are mapped into the shared contract. |
| Config/env -> provider factory | Environment values control provider selection and must be treated as authoritative but validated. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | T | `src/llm/config.ts` | mitigate | Validate provider keys against an allow-list so unknown values fail fast instead of routing to the wrong backend. |
| T-01-02 | Tampering | `src/llm/providers/lm-studio.ts` | mitigate | Keep request/response parsing inside the adapter and reject malformed payloads instead of passing them to the app layer. |
| T-01-03 | D | provider resolution | accept | Local LM Studio unavailability is an expected runtime failure for a v1 local-first setup; surface a clear error and let the caller handle it safely. |
</threat_model>
