<data_flow>
1. User submits topic + question count through the minimal trigger.
2. The trigger forwards the inputs directly to `generateQuiz` from `src/app/index.ts`.
3. The app layer performs input normalization, provider resolution, generation, and MCQ validation.
4. The trigger receives either validated MCQs or an app-safe error.
5. The rendering helper converts the result into displayable success or error markup.
6. The user sees rendered MCQs with question, four options, correct answer, explanation, or a clean app-safe error message.
</data_flow>

<ui_api_boundary_design>
- Choose one thin public boundary for v1: a single trigger module that owns the user interaction and delegates all generation work to `src/app/index.ts`.
- Keep the boundary input-only from the user perspective: topic + questionCount in, rendered result out.
- Do not add a second orchestration layer, client-side validation, or provider-specific branching.
- Keep the rendering boundary pure where possible so success/error display is easy to test and reason about.
- Treat the public app boundary as the only generation API; the trigger is presentation only.
</ui_api_boundary_design>

<error_display_strategy>
- Display app-safe errors verbatim at the boundary, but only the stable code/message returned by the app layer.
- Never surface raw provider or transport failures.
- Use a clear error state with concise text and no stack trace, retry hint, or provider details.
- Keep malformed/invalid model output indistinguishable from other app-safe failures to avoid leaking internals.
</error_display_strategy>

<scope_guardrails>
- No authentication.
- No persistence, saved history, or quiz scoring.
- No analytics.
- No additional providers.
- No retries or fallback logic.
- No app-layer validation/orchestration duplication.
</scope_guardrails>

<recommended_implementation_order>
1. Define the renderable view model and pure success/error rendering helpers.
2. Implement the minimal trigger that forwards topic/questionCount to `generateQuiz` and selects the render path.
3. Add end-to-end tests for the happy path and app-safe error display.
4. Add deterministic rendering tests for question/answer/explanation layout and error formatting.
</recommended_implementation_order>
