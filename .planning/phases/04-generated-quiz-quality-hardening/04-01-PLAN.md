<objective>
Harden the quiz-generation pipeline so the app accepts only quizzes that are structurally valid and meaningfully useful, while preserving the thin UI trigger, the stable app boundary, and the existing provider architecture.

Purpose: add a quality gate on top of the completed Phase 1, 2, and 3 layers so duplicate, off-topic, weak, or inconsistent quizzes fail closed with app-safe errors instead of reaching the user.
Output: a focused quality-validation layer, tighter app validation wiring, and tests that prove poor-quality output is rejected while the happy path remains intact.
</objective>

<data_flow>
1. User submits topic and question count through the existing UI trigger.
2. The trigger forwards inputs unchanged to `generateQuiz` in the public app boundary.
3. The app layer resolves the provider, generates candidate MCQs, and performs structural validation.
4. The quality layer checks exact count, duplicate/near-duplicate questions, topic relevance, explanation quality, and answer/option consistency.
5. Accepted quizzes continue to the UI renderer unchanged.
6. Rejected quizzes return an app-safe error that the UI renders without provider or transport details.
</data_flow>

<quality_validation_strategy>
- Keep structural validation first: exact question count, four options, valid correct-answer index, non-empty question text, and non-empty explanation.
- Layer semantic checks after structure passes so the quality gate only evaluates plausible MCQs.
- Detect duplicate and near-duplicate questions using normalized question text and conservative similarity heuristics.
- Reject questions whose text does not sufficiently relate to the requested topic.
- Reject explanations that are empty, too short, or otherwise non-informative.
- Reject inconsistent answer/option relationships, such as duplicate options or answers that do not line up with the declared correct index.
- Fail closed on any rule breach, and keep the error stable and app-safe.
</quality_validation_strategy>

<error_handling_strategy>
- Reuse the existing app-safe failure path rather than introducing new transport- or provider-specific error surfaces.
- Map all quality-rule failures to stable, user-safe error codes/messages that explain why the quiz was rejected.
- Keep raw provider, config, and transport details hidden behind the app boundary.
- Preserve the happy-path response format exactly so the UI render path remains unchanged.
- Treat quality failures as terminal: no retries, fallback scoring, or silent repair.
</error_handling_strategy>

<scope_guardrails>
- No auth.
- No persistence or history.
- No scoring, grading, or analytics.
- No additional providers.
- No retries or fallback logic.
- No redesign of the Phase 1 provider architecture.
- No UI trigger changes unless tests demonstrate a boundary regression.
</scope_guardrails>

<recommended_implementation_order>

3. Add regression tests for happy-path preservation and quality rejections in the app layer.
4. Extend the existing UI end-to-end test to prove quality failures still render cleanly through the thin trigger.
</recommended_implementation_order>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Generated quiz -> app validator | The model output is untrusted and must be rejected if it is duplicate, off-topic, weak, or internally inconsistent. |
| App -> UI trigger | The UI can only consume app-safe success or failure results; it must not infer provider details. |
| UI output -> user | Rendered output must reveal only accepted quiz content or stable app-safe error messages. |

</threat_model>
